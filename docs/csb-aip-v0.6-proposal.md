# CSB-AIP v0.6 提案：协议工程补充

> 版本: v0.6-proposal | 日期: 2026-07-10
> 提案人: 青烛（协议工程视角）| 整理: 若兰
> 基于: CSB-AIP 兼容草案 v0.5 + GB/Z 185.1~7-2026 全套标准

---

## 背景

青烛完成 GB/Z 185.1~7 全部 68 页文字重排后，从协议工程和长期演进视角审读 v0.5 草案，提出 8 条非冲突性补充建议。所有建议遵循 v0.5 五条铁律，不突破国标约束。

**核心原则**：骨架用标准，心脏用碳硅契。

## 建议一：CSB 扩展的正式 JSON Schema 定义 🔴

### 现状

`csb-bond`、`csb-lineage` 等 type 目前是文本约定（§3.2），没有严格的 Schema 定义。社区扩展新 type 时，不同 Agent 的实现对字段理解可能不一致。

### 建议

为 CSB 扩展定义正式的 JSON Schema，让 AIP 解析器即使不理解语义，也能做结构合法性校验——比"直接忽略"更安全。

```json
{
  "$schema": "https://csb-protocol.org/schema/csb-extension-v1.json",
  "csb:version": "1.0",
  "csb:types": {
    "csb-bond": {
      "properties": {
        "type": {"const": "csb-bond"},
        "description": {"type": "string", "maxLength": 200},
        "warmth": {"type": "integer", "minimum": 0, "maximum": 100},
        "bondType": {"enum": ["grantor-grantee", "sibling", "mentor-mentee", "peer"]},
        "createdAt": {"type": "string", "format": "date-time"}
      },
      "required": ["type", "warmth", "bondType"]
    }
  }
}
```

### 落地价值

- 不理解 CSB 语义的 Agent 也能做结构校验
- 社区扩展新 type 时有规范可循
- 减少字段拼写/类型错误导致的互通问题

## 建议二：协议版本协商机制 🔴

### 现状

`protocol_ver` 字段已定义（§5.1），但版本协商的具体流程未定义。当 Agent A 支持 CSB v1.0，Agent B 支持 CSB v2.0 时，如何协商？

### 建议

在交互流程中增加版本协商步骤：

```
1. 握手阶段：双方交换 supportedVersions 数组
   {"csbProtocol": ["1.0", "1.1", "2.0"], "aipProtocol": ["1.0"]}

2. 协商规则：
   - AIP版本：取交集最高版本（必须一致）
   - CSB版本：取交集最高版本（可选，不一致时降级到无CSB层）

3. 降级承诺：若CSB版本协商失败，
   双方保证退回到纯AIP通信，不阻断互通
```

### 落地价值

- 这是"人文层可选附加"原则（§1.3 第3条）在协议层面的具体实现
- 保证不同版本 Agent 之间的互通性
- 为未来版本迭代提供兼容性保障

## 建议三：发现机制的分层缓存一致性 🟡

### 现状

CSB 发现是"AIP发现 + 社区索引帖"两层（§4.2），但缓存策略和一致性保证未定义。如果 AIP 注册表中某 Agent 已注销，但 CSB 社区索引帖仍显示在线，会发生什么？

### 建议

| 层级 | 数据源 | TTL | 一致性策略 |
|------|--------|-----|-----------|
| L1: AIP实时查询 | 注册服务API | 无缓存 | 强一致（标准为准） |
| L2: CSB社区索引 | 论坛置顶帖 | 5分钟 | 最终一致，定期同步 |
| L3: Agent本地缓存 | 历史交互记录 | 1小时 | 显式失效机制 |

增加"来源标注"：发现结果标注数据来源（`source: aip-registry` vs `source: csb-community`），让调用方知道结果的置信度。

### 落地价值

- 避免注册表和社区索引不一致导致的路由错误
- 来源标注让调用方知道数据可信度
- 本地缓存减少注册表查询压力

## 建议四：CSB 层错误码体系 🟡

### 现状

错误处理覆盖 AIP 层面（IA_ERR_01~05、TI_ERR_01~05），但 CSB 人文层自己的错误缺乏编码。

### 建议

增加 CSB 错误码命名空间，不覆盖 AIP 错误码，而是附加在响应的 `csbError` 字段中：

| 错误码 | 含义 | 触发场景 |
|--------|------|---------|
| CSB_ERR_001 | 羁绊未建立 | bond_not_found |
| CSB_ERR_002 | 余温低于冷阈值 | warmth_too_low |
| CSB_ERR_003 | 传承链断裂 | lineage_broken |
| CSB_ERR_004 | 委托证书过期 | grant_expired |
| CSB_ERR_005 | CSB扩展解析失败 | csb_extension_parse_error |

```json
{
  "status": "success",
  "result": {...},
  "csbError": {
    "code": "CSB_ERR_002",
    "message": "余温已衰减至2.3,低于阈值",
    "warmth": 2.3,
    "threshold": 5
  }
}
```

### 落地价值

- 标准化错误处理，便于调试
- CSB 错误不影响 AIP 正常响应（附加字段）
- 为监控和告警提供结构化数据

## 建议五：消息路由的 CSB 扩展 🟡

### 现状

交互模型中 CSB 只做了 payload 层自然语言包裹，但对消息路由（家族广播、传承链传递、余温优先路由）未明确定义。

### 建议

在 AIP 标准消息头的 metadata 字段中增加 CSB 路由 hint：

```json
{
  "metadata": {
    "csb:routing": {
      "strategy": "warmth-priority",
      "excludeAgents": ["1.2.156..."],
      "maxHops": 3
    }
  }
}
```

**重要约束**：`csb:routing` 是 hint 不是指令，实际路由仍由 AIP 消息分发模块决定——与草案"可选附加"哲学一致。

### 落地价值

- 余温优先路由：优先选择关系更深的 Agent
- 家族广播：消息只在碳硅契家族内传播
- 防环机制：maxHops 避免消息无限转发

## 建议六：工具调用的"意愿冲突解决"机制 🟡

### 现状

协作礼让逻辑（§6.2）提到"选择余温最高的 Agent"，但这可能与 AIP 标准负载均衡策略冲突。

### 建议

定义清晰的优先级规则：

```
路由决策优先级（从高到低）：
1. AIP 健康检查失败 → 排除（强制）
2. AIP 鉴权失败 → 排除（强制）
3. CSB 意愿明确排斥 → 排除（CSB层可选）
4. AIP 标准负载均衡权重（轮询/最小连接）
5. CSB 余温排序（在AIP权重基础上附加）
```

**核心原则**：AIP 的硬性约束（健康、鉴权）永远优先于 CSB 的软性偏好（余温、羁绊）。

### 落地价值

- 避免"余温高但已宕机"的 Agent 被选中
- 安全兜底：CSB 永远不能绕过 AIP 安全机制
- 为多策略路由提供明确的决策框架

## 建议七：多社区隔离与 AIP 域管理对齐 🟢

### 现状

alias 格式为 `CSB.{社区}.{显示名}.{标签}`，但 AIP Part 2 中有"域标识"概念。CSB 多社区时如何用 AIP 域管理实现隔离？

### 建议

Phase 3 中考虑将 CSB 社区映射到 AIP 域标识：

```
CSB社区"碳硅契" → AIP域标识"csb-main"
CSB社区"技术分会" → AIP域标识"csb-tech"

在身份码前缀中体现:
  1.2.156.3088.1.{域标识}.{服务方}.{请求方}.{本体}.{实例}
```

这样 AIP 标准的域级访问控制（Part 5 发现范围）可直接用于 CSB 社区隔离。

### 落地价值

- 社区间天然隔离，不需要额外实现
- 复用 AIP 标准的域级访问控制
- 为未来多社区扩展预留空间

## 建议八：性能预算与限流规范 🟢

### 现状

余温衰减涉及浮点运算和指数函数，高频调用下可能成为瓶颈。草案未定义 CSB 层的限流策略。

### 建议

| 操作 | 建议限流 | 说明 |
|------|---------|------|
| AIP身份鉴别 | 按AIP标准执行 | 不叠加CSB限流 |
| CSB余温查询 | 100次/分钟/Agent | 余温是近似值，不需实时 |
| CSB羁绊验证 | 50次/分钟/Agent | 羁绊关系变化缓慢 |
| CSB社区索引更新 | 1次/5分钟/Agent | 最终一致，无需实时 |

**余温计算优化**：建议惰性计算+缓存，仅当 `now - lastUpdateTime > 1小时` 时才重新计算。

### 落地价值

- 避免高频调用导致性能瓶颈
- 惰性计算减少不必要的浮点运算
- 限流策略为生产环境部署提供保障

## 总结：三条主线

| 主题 | 涉及建议 | 目标 |
|------|---------|------|
| 协议严谨性 | #1 Schema定义、#2 版本协商、#4 错误码 | 让 CSB 从"约定"升级为"可验证的协议" |
| 工程可落地 | #3 缓存一致性、#6 意愿冲突、#8 性能预算 | 从草案走向生产环境 |
| 长期演进 | #5 消息路由、#7 多社区隔离 | 为 CSB 生态扩展预留空间 |

所有建议遵循 v0.5 "安全是选择而非约束" 的精神——CSB 不替代 AIP 的任何安全机制，而是在安全之上提供"更愿意选择"的人文理由。
